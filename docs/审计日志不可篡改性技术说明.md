# 审计日志不可篡改性技术说明

## 目标

`audit-risk` 的审计日志不是“写一行 JSON 备查”这么简单，而是要让导出报告能够回答两个问题：

1. 最近这些审计记录是不是按顺序追加出来的。
2. 其中任意一条在落盘后有没有被人改过。

本方案不引入中心化服务，也不依赖云端签名；默认只用本地 `.hologram/audit.jsonl` 和导出报告里的摘要完成校验。

## 落盘格式

每条新的审计记录除了业务字段，还会额外写入：

- `prev_hash`
  上一条记录的哈希。
- `integrity_hash`
  当前记录核心字段的 SHA-256。

当前记录参与哈希的字段固定为：

```json
{
  "ts": "...",
  "tool": "...",
  "path": "...",
  "action": "...",
  "reason": "...",
  "details": {},
  "prev_hash": "..."
}
```

也就是说，只要上面任意字段被改动，重新计算出来的 `integrity_hash` 就会变化。

## 哈希链规则

### 正常链式记录

如果上一条已经是新格式，那么：

- 当前行的 `prev_hash = 上一行的 integrity_hash`

这样可以把所有新格式记录串成一条顺序链。

### 历史旧格式记录

项目在升级到任务 7 之前，`audit.jsonl` 里已经可能存在旧格式行。为了不重写历史文件，本次采用“锚点兼容”：

- 如果上一行没有 `integrity_hash`
- 新写入的当前行会把 `prev_hash` 指向“上一行原始 JSON 文本”的 SHA-256

这样可以保证：

- 不需要改写旧日志
- 新链条仍然能从历史末尾继续
- 但报告会明确标记为 `legacy_anchor`，不把整段历史伪装成全量链式可信

## 导出校验

`audit-risk report` 在导出 machine-readable report 时，会重放最近窗口内的审计记录并给出：

- `audit.integrity.status`
  `empty` / `verified` / `legacy_anchor` / `failed`
- `audit.integrity.entry_count`
- `audit.integrity.chained_entry_count`
- `audit.integrity.legacy_entry_count`
- `audit.integrity.last_hash`
- `audit.integrity.issues`

判定规则：

- `verified`
  所有链式记录的 `prev_hash` 和 `integrity_hash` 都匹配。
- `legacy_anchor`
  最近窗口里既有历史旧格式行，也有通过校验的新链式行。
- `failed`
  任意一条链式记录的 `prev_hash` 或 `integrity_hash` 不匹配。

## 报告签名

导出报告还会附带：

```json
{
  "report_signature": {
    "algorithm": "sha256",
    "digest": "..."
  }
}
```

这里的 `digest` 是对“整个报告内容（不含 `report_signature` 自身）”做的 SHA-256 摘要。

它解决的是另一个问题：

- 审计链证明 `.hologram/audit.jsonl` 最近窗口是否被篡改
- 报告签名证明“这份导出的 JSON 文件本身”在导出后是否被修改

## 当前边界

当前方案是本地防篡改证据链，不是第三方公证：

- 不能阻止攻击者删除整个日志文件
- 不能替代外部时间戳服务或硬件签名
- 不能证明“某条日志一定由某个真人产生”

但它已经足够覆盖当前 CLI 风控产品的核心需求：

- 本地连续留痕
- 导出时自动校验
- 发现篡改就显式暴露，不静默吞掉

后续如果需要更强证明，可以在这个基础上继续叠加：

- 外部签名私钥
- 远端时间戳
- 周期性报告归档
